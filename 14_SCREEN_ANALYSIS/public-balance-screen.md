# Screen Analysis — Public Coin Balance Check (`/check-balance`)

**File:** `src/features/public/container/PublicBalanceScreen.tsx`
**Styles:** **inline `style={{...}}` objects throughout — no CSS Module file for this screen** (contrast with its sibling `PublicSubmitScreen`, which does use a `.module.css`). All styling lives directly in the TSX.
**Route:** `path: "/check-balance"` (`id: "check-balance"`, `src/routes/routes.config.ts:55-56`), rendered directly in `src/routes/AppRouter.tsx:74` (`"check-balance": <PublicBalanceScreen />`) — no dynamic slug param (org context, if any, is resolved server-side from the verified email, not the URL).

---

## ⚠️ Design-system boundary — read this first

**This screen is the second of exactly two public, unauthenticated, QR-code-landing screens in the product** (alongside `PublicSubmitScreen`), and — like its sibling — it **deliberately does not use the internal app's design system** (`src/shared/components/ui/index.tsx`, `.ai-design-dna/12_DESIGN_TOKENS`). Where `PublicSubmitScreen` packages its own mini design system into a `.module.css`, **this screen goes a step further and inlines every style as a JS object literal directly on each element** — there is no shared stylesheet backing it at all, not even a screen-local one. Treat the values below as **specific to this one file**, not as tokens to look up in `12_DESIGN_TOKENS/*.json` (they mostly aren't there) and not as a pattern to copy into `PublicSubmitScreen.module.css` either — the two public screens are visually siblings (same ink/white palette, same rough geometry) but are implemented with two different, non-shared styling mechanisms.

Concretely:

- **Every color, size, radius, and shadow is a literal inline value** — e.g. the primary button is `height: 50` / `borderRadius: 999` / `background: "#0A0A0A"` written directly at the call site (lines 503-521), not a class, not a token reference, not a shared constant. The OTP step's card is `border-radius: 20`, `boxShadow: "0 4px 24px rgba(10,10,10,0.06)"` (lines 546-550) — again, bespoke to this file.
- **No shared `Button`/`Field`/`Input` import.** The only shared imports are `LanguageSwitcher` and a date-formatting helper (`formatDate` from `@/lib/utils` — note: the **legacy** `src/lib` path, not the canonical `src/shared/lib`, per the architectural-debt note in `.ai-design-dna/00_PROJECT_IDENTITY.md`; flagged as observed, not fixed).
- **Its own spinner:** a bespoke `Spinner` component (lines 57-73) built from a `motion.span` with a manually-drawn CSS border-circle (`border: "2px solid rgba(255,255,255,0.3)"`, `borderTopColor: "#fff"`), animated via a raw Framer Motion `rotate: 360` / `duration: 0.75` / `ease: "linear"` / `repeat: Infinity` — a completely different implementation from `PublicSubmitScreen`'s CSS `@keyframes spin` *and* from whatever `Spinner` primitive (if any) the internal app's shared library defines. Three different spinner implementations likely coexist in this repo; this document does not reconcile them.
- **Its own OTP input:** a hand-rolled 6-box digit input (`OtpInput`, lines 77-163) with inline `onFocus`/`onBlur` handlers that mutate `e.target.style` directly (imperative DOM style mutation inside React event handlers, lines 151-158) — a pattern not seen anywhere else described in this design-DNA pack, and notably *not* using React state/CSS classes for the focus visual, unlike literally every other focus-state implementation in the codebase (both public screens' CSS-based `:focus` and the internal app's Tailwind `focus:` classes).
- **Signature radial-gradient glow backgrounds** — `radial-gradient(circle, #FFF8E1 0%, #FEF3C7 60%, transparent 100%)` (warm amber, top-right, lines 348-361) and `radial-gradient(circle, #F3F4F6 0%, #E5E7EB 60%, transparent 100%)` (cool gray, bottom-left, lines 362-375) — a decorative technique that appears **nowhere else in the product**: the internal app uses flat surfaces/hairline borders/tinted-ink shadows, and even `PublicSubmitScreen` only uses flat low-opacity ink circles (`rgba(10,10,10,0.02-0.03)`), not gradients. This is the single most distinctive, non-transferable visual signature of this screen.

**Directive for any AI extending this codebase:**
- If extending the **internal admin app** — never reach for this file's inline-style patterns, its imperative `e.target.style` focus handling, its radial-gradient glows, or its bespoke `Spinner`/`OtpInput`. Use `src/shared/components/ui/index.tsx` and the documented token files.
- If extending **this screen or `PublicSubmitScreen`** — recognize they are *not* built the same way even though they look related. Do not assume a class defined in `PublicSubmitScreen.module.css` is available here (it isn't — this file has no CSS Module import at all), and do not "helpfully" migrate this screen's inline styles into that module or vice versa without being asked; document/extend each in its own existing mechanism.
- Do not treat the amber/gray radial-gradient glow as a reusable brand element to sprinkle elsewhere — it is scoped to this one balance-check screen's visual identity (coins/rewards context) and has no presence anywhere else in the app, including `PublicSubmitScreen`.

## 1. Purpose & access

- **Purpose:** Lets an anonymous employee check their accumulated "innovation coin" balance, earn/redeem totals, and transaction history by verifying ownership of a work email via a one-time 6-digit code — no password, no persistent login, no link back into the authenticated app.
- **Access:** Fully public, unauthenticated route (`/check-balance`), reached the same way as `PublicSubmitScreen` — via a QR code, independent of any organization slug in the URL (the org context is implicit server-side, tied to whichever tenant the verified email belongs to).
- **Ephemeral session model:** there is no persisted auth token stored client-side after verification — `balanceData` lives only in React state; a page refresh returns the visitor to the email step. This is a deliberately lightweight, "look-up-and-leave" utility, not a login.
- A "Check another account" reset button (line 1074) explicitly supports the "walk-up kiosk" use case — multiple different employees using the same device/QR code in sequence.

## 2. Layout

- Full-viewport, centered, single-column card layout: `minHeight: "100vh"`, `flexDirection: "column"`, `alignItems/justifyContent: "center"` (lines 329-342) — content is vertically and horizontally centered regardless of step, unlike `PublicSubmitScreen`'s top-anchored, bottom-progress-dot layout.
- Content column width is **step-dependent and animates**: `maxWidth: step === "balance" ? 480 : 400` with `transition: "max-width 0.3s ease"` (lines 380-385) — the card visibly widens when the data-dense balance view replaces the narrow email/OTP steps, a subtle but deliberate density-driven layout adaptation not seen in `PublicSubmitScreen` (which holds a fixed 480px column across all steps).
- Two fixed-position decorative radial-gradient circles anchor opposite corners (top-right warm amber, bottom-left cool gray) — same conceptual role as `PublicSubmitScreen`'s `.bgShape1`/`.bgShape2` but rendered as gradients rather than flat tints, and using `position: "fixed"` rather than `absolute` (so they don't scroll with content, though the page itself has no scroll).
- Step 1 (email) renders "bare" — no card border/background, just centered text and controls directly on the white page background. Steps 2 (OTP) and 3 (balance) both wrap their content in an explicit white card with `border: "1px solid #E5E7EB"`, `borderRadius: 20`, `boxShadow: "0 4px 24px rgba(10,10,10,0.06)"` — a visual escalation from "bare landing" to "contained document" as the flow progresses and more information needs framing.
- `LanguageSwitcher` fixed top-right, identical positioning convention to `PublicSubmitScreen`.

## 3. Hierarchy

- Step 1 leads with a large emoji glyph (🪙, 52px, via the `CoinIcon` helper, lines 167-180) as the top-of-page focal point — the only screen in the product's public flow that uses an emoji as primary iconography rather than a hand-drawn SVG or the Δ wordmark.
- Title/subtitle pairing mirrors `PublicSubmitScreen`'s pattern (24px bold ink title, 14px gray `#6B7280` subtitle) but **without** the mono uppercase eyebrow label that precedes titles in `PublicSubmitScreen` — this screen skips that eyebrow tier entirely on step 1 and 2.
- On the balance step, hierarchy is built almost entirely by **size**, not just weight/color: the current-balance number is rendered at 44px/700 (`letter-spacing: -0.03em`) — the single largest piece of text anywhere across both public screens — with its "pts" unit deliberately smaller (20px, gray) beside it on the same baseline (`alignItems: "baseline"`, lines 820-844), a classic "big number, small unit" stat-tile convention.
- Secondary stats (Earned/Redeemed) step down to 22px numbers inside color-coded cards, and individual transaction-row amounts step down further to 13.5px mono — a clear three-tier numeric hierarchy: 44px hero stat → 22px summary stats → 13.5px line-item stats.
- User identity (avatar + name + email) is given top billing on the balance card (first thing rendered, lines 738-800) before any numbers — "whose balance is this" is established before "what is the balance."

## 4. Spacing

- Step 1: `32px 16px` outer page padding (tighter horizontal padding than `PublicSubmitScreen`'s `24px`, since this screen has no persistent side chrome); title-to-subtitle `10px`, subtitle-to-input `32px` gap (via bottom margin on the subtitle).
- OTP card: `36px 32px` internal padding, `20px` gap before the OTP boxes, `20px` after them, `16px` bottom margin under the verify button before the resend row.
- Balance card: `32px 32px 28px` internal padding (deliberately less bottom padding, since the reset button sits flush near the bottom edge); `24px` after the identity block, `20px` after the balance hero number, `20px` after the earned/redeemed grid, `20px` after the transaction list — a consistent ~20-24px rhythm between major content blocks within the card.
- Individual transaction rows: `11px 14px` padding, `1px solid #F3F4F6` hairline dividers between rows (not the `#E5E7EB` used for the outer card border) — a deliberately lighter internal divider than the container border, standard "nested hairline is lighter than outer border" convention also seen in the internal app's tables.
- OTP digit boxes: `8px` gap between the six boxes (`justifyContent: "center"`, line 122).

## 5. Typography

- No mono-font eyebrow labels at all on steps 1-2 (unlike `PublicSubmitScreen`'s "STEP 1 OF 2"), but `JetBrains Mono` reappears heavily on the balance step for exactly the data-oriented roles the internal app reserves it for: the "CURRENT BALANCE"/"EARNED"/"REDEEMED"/"HISTORY · N TRANSACTIONS" uppercase eyebrow labels (11px, `letter-spacing: 0.1em`), the OTP-destination email echo (`"JetBrains Mono, monospace"`, line 617), each transaction's idea-ref code, and every point-value/amount (`±N`, `44px`/`22px`/`13.5px` numbers) — i.e. **any number or identifier gets mono, any prose gets the default sans**, the exact same rule the internal design system documents, independently re-derived here in inline styles.
- The OTP digit boxes themselves use `JetBrains Mono, monospace` at 22px/600 with `caretColor: "transparent"` (line 149) — a deliberate choice to hide the text caret so the box reads as a static digit slot rather than a text field, a small but considered typographic/UX detail.
- Scale: 44px hero balance number (largest text in either public screen) → 24-26px would-be title tier but step 1's title is actually 24px, OTP step's title is smaller at 20px (a step-appropriate de-escalation, since the OTP screen is a "checkpoint" not a "destination") → 22px summary stat numbers → 17px user name → 14-15px body/buttons → 13.5px secondary numbers/labels → 12-12.5px helper/meta text → 11px mono eyebrows.
- Negative letter-spacing on numerals (`-0.02em` to `-0.03em`) mirrors the same "tight tracking signals emphasis" rule used throughout both public screens and the internal app.

## 6. Components used

Entirely local/inline — no shared library components beyond `LanguageSwitcher`:

- **`Spinner`** (lines 57-73) — bespoke Framer-Motion-driven rotating ring, white-on-transparent, sized for use inside a dark button (18px, semi-transparent white ring with a solid white leading edge).
- **`OtpInput`** (lines 77-163) — 6 independent single-character `<input>` boxes with auto-advance-on-type, backspace-to-previous-box, and full paste-splitting support (`handlePaste`, lines 111-119, strips non-digits and distributes across boxes). Focus/blur visuals are applied by direct `element.style` mutation in the event handler rather than CSS `:focus` — an outlier pattern versus every other focus treatment in the app.
- **`CoinIcon`** (lines 167-180) — a single 52px emoji (🪙) with `userSelect: "none"`, functioning as the step 1 hero graphic.
- **Pill-shaped email input** (step 1) — 50px height, `borderRadius: 999`, center-aligned text, red border swap on `emailError`; note this is a **plain pill**, distinct from `PublicSubmitScreen`'s rectangular 12px-radius `.input` — the two public screens do not even share the same input geometry with each other.
- **Pill primary buttons** — "View my coins" (step 1, 50px) and "Verify & view balance" (step 2, 46px) both `borderRadius: 999`, black fill, swapping to `Spinner` + a "…ing" label while loading, and using `opacity` (0.5-0.8) rather than a background-color change to signal disabled state — different disabled-state mechanism from `PublicSubmitScreen`'s `.primaryBtn:disabled` (which changes background to `rgba(10,10,10,0.55)`).
- **Resend-code text button** — bare inline button, gray when on cooldown (`#9CA3AF`) vs. ink-black when active, with the countdown ("Resend in {{seconds}}s") interpolated directly into the label rather than shown as a separate timer element.
- **Stat cards (Earned/Redeemed)** — two color-themed cards in a 2-column grid: green triad (`#F0FDF4` bg / `#BBF7D0` border / `#16A34A`-`#15803D` text) for Earned, red triad (`#FEF2F2` bg / `#FECACA` border / `#DC2626`-`#B91C1C` text) for Redeemed — this is the **same semantic red/green-family logic** the internal app uses for status badges, but with bespoke hex values local to this screen rather than the shared `12_DESIGN_TOKENS/semantic-colors.json` triads (worth cross-checking for drift if this screen is ever revisited).
- **Transaction history list** — a bordered, rounded, internally-scrolling (`maxHeight: 260, overflowY: "auto"`) list of rows, each with a green/red status dot, a truncated title (idea title → description → type-label fallback chain, line 1005-1007), a metadata line (type label · idea ref · formatted date), and a right-aligned mono signed point delta — functionally analogous to the internal app's list-row patterns (dot + title + meta + trailing value) though implemented with zero shared code.
- **Avatar** — 52px circle, either the user's `profilePicture` image (`objectFit: "cover"`) or a generated initials fallback (ink-black bg, white initials) via the local `initials()` helper (lines 51-53) — same visual idea as any avatar component elsewhere in the app, independently implemented.
- **Reset button** ("Check another account") — full-width, 44px, light-gray fill (`#F3F4F6`) rather than black, `999px` radius — the one button on this screen that is intentionally low-emphasis (gray, not black) since it's a "start over" action, not a forward-progressing CTA.

## 7. Interaction (OTP/QR flow)

- **Three-step linear flow:** `email` → `otp` → `balance`, state-machine driven exactly like `PublicSubmitScreen` (`step` state + `advance()` helper + Framer Motion direction flag), but with **no visible progress indicator** at all (no dots, no "step X of Y" label) — the visitor has no explicit sense of how many steps remain, a deliberate-or-incidental simplification versus `PublicSubmitScreen`'s explicit 2-dot progress bar.
- **Step 1 (email):** single email input, `onKeyDown` Enter-to-submit shortcut (line 438, not present on `PublicSubmitScreen`'s inputs), client-side regex validation identical in pattern to `PublicSubmitScreen`'s (`/^[^\s@]+@[^\s@]+\.[^\s@]+$/`) but duplicated locally rather than shared. On submit, POSTs to `/coins/request-otp`; success starts a 60-second resend cooldown (`setResendCd(60)`) and advances to the OTP step.
- **Step 2 (OTP):** 6-digit auto-advancing input, a "Back" link that resets `step` to `"email"` via a `setTimeout(..., 0)` deferred state update (line 555 — deferring the step change by a tick after setting direction, likely to let the exit-direction prop apply before the AnimatePresence key changes; a subtle implementation detail worth preserving if this flow is ever touched), inline countdown-gated "Resend code" link, and a verify button disabled until all 6 digits are entered (`otp.length < 6`). Verify POSTs to `/coins/verify-otp`; success stores the full `BalanceData` payload and advances to `balance`.
- **Step 3 (balance):** purely a read-only data display — no further form interaction except the bottom "Check another account" reset, which fully clears all state (`handleReset`, lines 318-326) and returns to step 1.
- **Transition animation:** same `AnimatePresence mode="wait"` + horizontal slide + fade recipe as `PublicSubmitScreen`, but with a shorter travel distance (`x: ±60%` vs. `±100%`) and a stiffer spring (`stiffness: 400, damping: 38` vs. `380`/`36`) — a subtly snappier feel, consistent with this screen's more "utility lookup" tone versus the more ceremonial submission flow.
- **Staggered reveal on the balance card:** identity block (delay 0.04s) → balance hero (0.08s) → earned/redeemed grid (0.14s) → transaction history (0.2s) → reset button (0.28s) — the same "cascading fade+slide-up" choreography technique as `PublicSubmitScreen`'s success screen, reused here for a data reveal rather than a celebration.

## 8. Accessibility

- The email input's error styling swaps border color to `#DC2626` and the error message includes an inline warning-circle SVG + text (lines 470-498) with `role`/`aria-live` absent — same gap pattern as `PublicSubmitScreen`.
- The 6-box OTP input has no `aria-label` per box (e.g. "digit 1 of 6") and no `aria-describedby` linking it to the error message below — a screen-reader user would encounter six unlabeled text inputs in sequence. `caretColor: "transparent"` (line 149) additionally removes a normally-available visual affordance (the blinking caret) without providing an alternative "current position" cue for low-vision users beyond the browser's default focus ring behavior triggered by the manual `onFocus` style mutation.
- Imperative `e.target.style` focus/blur handling (lines 151-158) means the focus ring is applied only via a hard-coded `boxShadow`, not a semantically-meaningful `:focus-visible`-driven style — functionally fine for mouse/touch, but bypasses the platform's native focus-visible heuristics that distinguish keyboard from pointer focus.
- The resend-code and "check another account" buttons correctly use native `disabled` attributes tied to real state (`resendCd > 0 || loading`), which does convey disabled state to assistive tech correctly, unlike the purely-visual required-field asterisks on `PublicSubmitScreen`.
- No `prefers-reduced-motion` accommodation here either — consistent gap with `PublicSubmitScreen`.
- Color contrast: green (`#15803D` on `#F0FDF4`) and red (`#B91C1C` on `#FEF2F2`) stat-card text both comfortably clear WCAG AA; the `#9CA3AF` meta text at 11-13px is the same borderline-but-consistent secondary tier as elsewhere in the product.

## 9. Data density

- **Bimodal density**, unlike `PublicSubmitScreen`'s uniformly low density: steps 1-2 are extremely sparse (a single input, or six digit boxes, centered in generous whitespace), while step 3 (balance) is comparatively dense for a public screen — a stat hero, two summary cards, and a scrollable list of arbitrarily many transaction rows all inside one 480px-wide card with an internal `maxHeight: 260` scroll region.
- The transaction list's internal scroll (rather than paginating or truncating) is the one place this screen approaches "internal app" information density, though still far lighter than any authenticated dashboard table (no column headers, no sort/filter, no row actions — purely a read-only ledger).

## 10. Reusable ideas

- **Step-dependent animated container width** (`maxWidth` keyed off `step`, with a CSS `transition`) is a nice small technique — worth considering for `PublicSubmitScreen` too, which currently holds a fixed 480px width across a form step and a (visually simpler) success step.
- **Enter-to-submit on the email field** (line 438) is a small usability win `PublicSubmitScreen` lacks on its equivalent name/email inputs — worth propagating to that screen if it's ever revisited, staying within its own CSS Module rather than borrowing this screen's inline-style mechanism.
- **Paste-splitting OTP input** (`handlePaste`, lines 111-119) is a well-built, self-contained pattern (strip non-digits, slice to 6, distribute, focus the right next box) that could be lifted almost as-is into any other OTP-style flow this product adds later — but if promoted to a shared component, it should be rebuilt to use CSS `:focus` classes instead of the current imperative `element.style` mutation, and given proper per-box `aria-label`s.
- **Semantic green/red stat-card pairing** (Earned/Redeemed) is a clean, reusable *visual idea* (colored-tint card, matching border, matching text color, uppercase mono eyebrow) — but do not literally reuse these exact hex values without checking them against `.ai-design-dna/12_DESIGN_TOKENS/semantic-colors.json`, since they appear to be locally authored rather than pulled from that token file, and may already have drifted from it.
- **Do not port:** the radial-gradient glow backgrounds, the imperative-style-mutation focus handling, or the plain-inline-style-only approach (no CSS Module, no shared classes) into `PublicSubmitScreen` or the internal app. If this screen is ever refactored, the more consistent move is to give it its own `.module.css` (matching `PublicSubmitScreen`'s mechanism) rather than importing internal-app tokens/components.
