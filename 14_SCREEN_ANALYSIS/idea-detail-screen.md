# Idea Detail Screen

Source of truth: `src/features/ideas/container/IdeaDetailScreen.tsx` (full file, 1022 lines). Supporting components in `src/features/ideas/components/*` and one legacy import from `src/components/*`. Tokens cited from `.ai-design-dna/02_DESIGN_DNA.json` and `12_DESIGN_TOKENS/*.json`.

## Purpose & Access

Deep-dive record view for a single idea: `/ideas/:ideaId` (routed via `IdeaDetailScreen`, wrapped in `Shell`). It is where an idea actually gets *worked* — stage progression, evaluation questionnaires, financial impact capture, document exchange, threaded remarks, and an audit-style activity feed all live here, as opposed to the Kanban board (`KanbanScreen.tsx`) which is the move/triage view.

Role-gated rendering, driven by `useAuthStore((s) => s.role)`:

- **manager** (Innovation Manager) — full read/write. Sees `FinancialImpactCard`, `IdeaTodoCard` (private to-do), can edit stage questionnaires (`DynamicStageQuestionnaires` `isDisabled` becomes `false` only when `role === "manager"`), and gets the header action cluster (Reject / Approve / Ready for Pitch).
- **management** — read-mostly reviewer. Header action buttons (`Reject Idea`, `Approve Idea`, `Ready for Pitch`) are entirely omitted (`role !== "management" && idea` gate on the `Shell actions` prop). Cannot see or edit the submitter identity block (see privacy rule below). Can still upload documents, post remarks/mentions, and view the questionnaire answers read-only.
- Any other authenticated role — same as management for the identity gate (the check is `role !== "management"`, not an allow-list), but does get the header action buttons since that gate is specifically `role !== "management"`.

Access to the page itself requires `isAuthenticated` (queries are gated with `enabled: isAuthenticated && !!ideaId`); there is no separate route guard visible in this file — gating is entirely role-conditional rendering inside one component, not a redirect.

## Layout

Two-tier vertical layout inside `Shell` (crumbs: "Ideas Board" → idea title):

1. **Stage stepper strip** — full-width card (`bg-card border border-border rounded px-5 sm:px-8 py-2 sm:py-4 mb-[22px]`) containing `WorkflowStepper`, horizontally scrollable (`overflow-x-auto`) for narrow viewports even though this is a desktop-only app.
2. **Two-column body** — `grid grid-cols-1 lg:grid-cols-[1.65fr_1fr] gap-6 items-start pb-12`, collapsing to a single stacked column below `lg`.
   - **Main column (1.65fr, `flex flex-col gap-6`)**:
     - Details/Attachments card — internal `Tabs` (not page-level tabs) switching between a "Details" panel (summary/problem text + `DynamicStageQuestionnaires`) and an "Attachments" panel (uploaded-file chip list), with an `UploadDocumentModal` trigger and a conditional `Save` button (only visible on the Details tab, only enabled for `manager` role with unsaved changes).
     - `FinancialImpactCard` — manager-only, rendered as a sibling card below the details card, not inside it.
   - **Right rail (1fr, `flex flex-col gap-5`)**:
     - `<details>`-based collapsible "Submission Information" panel (native disclosure element, default `open`), containing reference/submitted/status/tag rows and, gated by the privacy rule, submitter + assigned-manager rows.
     - `IdeaTodoCard` — manager-only private to-do list.
     - Activity/Remarks card — internal `Tabs` switching between an `ActivityFeedRow` list (scrollable, `max-h-[360px]`) and a threaded remarks list + `MentionInput` composer (also `max-h-[360px]` scroll region).
     - `AwardedPointsCard` — another native `<details>` disclosure, per-stage point ledger + total.
3. Three screen-level overlays rendered outside the grid: `EmployeeIdeasDialog` (mention/avatar click-through profile), a confirm `Modal` for "Mark as Ready for Pitch," and `DocumentPreviewModal`.

The loading state swaps the entire body for `IdeaDetailSkeleton` (own skeleton markup, not a shared `Skeleton` primitive) while `idea` is being fetched; an idea-not-found state renders plain centered text instead of the two-column grid.

## Hierarchy

- Page-level: `Shell` crumbs establish "Ideas Board → {idea title}"; `Shell`'s `actions` slot carries the primary CTA cluster (danger Reject, success-tinted Approve, primary Ready-for-Pitch) or a static rejected/approved status pill when the idea is in a terminal state — i.e., the header itself communicates idea status via color before the user reads anything else.
- Card-level: every panel uses the same `bg-card border border-border rounded` shell with either a `Tabs` header or a `<details><summary>` disclosure header — no page uses a plain static heading; every section is either tabbed or collapsible, keeping the page dense but keyboard/click-navigable.
- Two of the right-rail cards (`Submission Information`, `Awarded Points`) are native `<details>` elements rather than the shared `Modal`/`Tabs` components used elsewhere — a second pattern for "collapsible section" living alongside the Tabs pattern in the same screen.
- Typographic hierarchy inside cards is by size + color, not weight jumps: card titles ~13.5px/600, labels 12.5px `text-text-2`, body 13–13.5px, meta/timestamps 11–12px `text-text-3`. No H1/H2 tags are used inside the content area — the `Shell` crumbs are the only true heading.

## Spacing

- Outer card gap: `gap-6` (24px) between stepper strip and body grid, and between right-rail cards.
- Card body padding: `p-[22px]` for the details card, `p-[5px_20px]` for `<details>` info rows, `px-6 py-5` for the activity/remarks card — three different padding values for visually similar card types (not a single "card body" token reused consistently).
- `InfoRow` (local helper, not from the shared catalog) uses `py-2.5` per row with a `border-b border-border` hairline divider — matches the DNA's row-divider convention seen in `IdeaRow` (`13_COMPONENT_CATALOG/idea-row.md`).
- Document chips: `pl-2.5 pr-2 py-1.5`, `gap-2` between chips (`flex flex-wrap gap-2`).
- Activity/remarks lists scroll-cap at `max-h-[360px]` with `pr-1.5` reserved for the scrollbar gutter (class `scroll-area`, presumably a custom scrollbar treatment defined globally, not inline here).
- Micro-spacing throughout uses non-integer Tailwind arbitrary values (`px-[8px]`, `mb-[10px]`, `p-[11px_13px]`) rather than the standard spacing scale steps documented in `12_DESIGN_TOKENS/spacing.json` — this screen leans on one-off pixel values more than the token file's `mostFrequentSteps` list suggests is canonical.

## Typography

- Section labels: `text-[12.5px] text-text-2` (matches DNA `typography.scale.label`).
- Body/description text: `text-[13.5px] leading-[1.7]` for the Summary/Problem blocks — a wider line-height than the DNA `typography.scale.body` default, presumably for long-form readability.
- Card/section titles: `text-[13.5px] font-semibold` (Submission Information, Awarded Points summaries) — matches `typography.scale.sectionCardTitle` (13.5px/600) from `12_DESIGN_TOKENS/typography.json`.
- Remarks/activity author name: `text-[13px] font-semibold`; remark body `text-[13px] leading-[1.55]`.
- Tag chips: `text-[11.5px] font-semibold`, contrast-computed per chip via the local `tagTextColor()` helper (relative luminance threshold 0.58) — a bespoke, file-local implementation rather than a shared token/utility, duplicating logic that likely exists elsewhere for tag rendering.
- Modal title (Ready-for-Pitch confirm): `text-[16px] font-semibold text-[#0A0A0A]` — a hardcoded hex instead of the `text-text`/ink token class used elsewhere in the same file, a small consistency gap.

## Interaction

- **Stage stepper (`WorkflowStepper`)**: pure, read-only progress visualization — filled circle + checkmark for completed stages, bold-outlined circle with stage number for the current stage, faded (`opacity-50`) for upcoming stages, animated connector fill (`scale-x` transition, 400ms) between steps. It does not itself move the idea between stages; stage movement happens via the Kanban board's drag-and-drop / `StageMovementModal` (legacy `src/components/modal/StageMovementModal.tsx`, not used on this screen) or via the questionnaire "Save" action indirectly advancing eligibility.
- **Stage evaluation forms (`DynamicStageQuestionnaires`)**: renders one form block per stage that (a) has questions defined on `stage.evaluationForm.questions` and has been reached (`stage.order <= currentStage.order`), or (b) already has a submitted evaluation (including "orphan" evaluations whose stage no longer matches, kept visible so history isn't lost). Supports text, textarea, number, email, select (via shared `Dropdown`), radio, multiselect (checkbox group), and file question types. Dirty-tracking is done by deep-comparing current answers against an `initialAnswersRef` snapshot (order-insensitive for multiselect arrays); dirty state and saving state are bubbled up via `onDirtyChange`/`onSavingChange` callbacks so the *screen's* header-adjacent Save button (not a button inside the component) can be enabled/disabled. Save is exposed as an imperative `ref.save()` (via `useImperativeHandle`/`forwardRef`), invoked from the parent's toolbar button — an unusual inversion where the child owns the data but the parent owns the trigger UI. Save uploads any staged files first (`POST /ideas/:id/stages/:stageId/evaluate/upload`), then submits all dirty stages' responses (`POST /ideas/:id/stages/:stageId/evaluate`), stage-by-stage, only for stages that are actually dirty. Editing is gated by `isDisabled = isActionDisabled || role !== "manager"` — management and any non-manager role always see the forms as read-only.
- **Document upload (`UploadDocumentModal`)**: staged, multi-file, drag-and-drop dropzone with a 10-file total cap (existing + staged) and a 20MB per-file size cap, both enforced client-side with toast feedback before any network call. Files are held as blob URLs (`URL.createObjectURL`) for inline thumbnail/preview until "Upload" is confirmed, at which point the parent's `uploadDocsMutation` POSTs each file individually via `FormData` (`Promise.all`), then switches the details card to the Attachments tab and refetches the file list. Cancelling revokes all staged blob URLs to avoid leaks.
- **Document preview (`DocumentPreviewModal`)**: branches on file category (`getFileCategory`) — native `<img>` for images, native `<video controls>` for video, native `<audio controls>` for audio, and `react-doc-viewer` for everything else (PDF/Office/etc.), with its own header disabled (`disableHeader`) in favor of the modal's own title bar and a Download button.
- **Document removal**: routed through `useUiStore().openModal("delete-document", {...})` — i.e. a confirm dialog is owned by the global modal registry, not local state, consistent with the Reject/Approve/Reassign/Tag-management actions on this screen which all go through the same `openModal(key, payload)` pattern rather than local `useState` booleans (the Ready-for-Pitch confirm is the one exception, using local `showPitchConfirm` state and the shared `Modal` directly).
- **@-mentions (`MentionInput` legacy + `MentionBadge`)**: `MentionInput` is a hand-rolled `contentEditable` rich editor (not a textarea) that detects an in-progress `@token` by walking back from the caret, shows a floating fixed-position suggestion dropdown (repositioned on scroll/resize), supports arrow-key navigation + Enter-to-select + Escape-to-close, and serializes selected mentions into inline `<span data-mention-id>` chips that round-trip to a `@[id:Name]` string token on submit. Rendered mention text elsewhere (activity feed rows, remark history) is parsed back out of that `@[id:Name]` token via a shared regex (`/(@\[[^\]]+\])/g`) and rendered as `MentionBadge`, which shows a hover tooltip (name + email) via `createPortal` to `document.body` and is clickable to open `EmployeeIdeasDialog` (via `profileEmployee` state lifted up to `IdeaDetailContent`). The remark composer explicitly labels itself "Visible to reviewers only," and is hidden entirely when `isActionDisabled` (idea already rejected or approved by management).
- **Activity feed (`ActivityFeedRow`)**: read-only chronological list, one colored icon-badge per `ActivityType` (submitted/stage_moved/reviewer_assigned/remark_added/file_attached/todo_created/evaluation_submitted/presentation_created/idea_rejected, plus a generic globe fallback), inline mention parsing on the description text, and type-specific secondary content (a quoted preview box for `remark_added`, a red reason box for `idea_rejected`).
- **Remarks tab**: separate from Activity — shows `idea.reviews` (not the activity feed) as a chat-like list with avatar + name + timestamp + mention-parsed body, then the `MentionInput` composer below. Posting calls `POST /ideas/:id/review` and invalidates the `idea` query (not a dedicated `reviews` query — the whole idea record is refetched).
- **Header actions**: Reject (danger button, opens `reject-idea` modal), Approve (only shown when `idea.isReadyToApprove` **and** `role === "manager"`; management/other roles never see Approve even though they might see Reject — Approve is a manager-only escalation of the reject/pitch cluster), and Ready-for-Pitch (disabled with a tooltip explaining why — rejected idea, already ready, or hasn't reached the final stage yet — computed from `stages` sorted by `order`).
- **Tag management, reassignment, submitter change**: all deferred to `openModal("tag-management" | "reassign" | "change-submitter", { idea })` — this screen only renders the trigger affordances (pencil/tag icons), not the modal bodies themselves.

## Accessibility

- Two right-rail sections use native `<details>/<summary>` for disclosure, which gives free keyboard operability (Enter/Space to toggle, native focus handling) without custom ARIA — a good pattern, but it's inconsistent with the `Tabs` component used for the other two card headers on the same screen, so keyboard behavior differs section to section (tab-list keyboard model vs. native disclosure model).
- `MentionInput` is a fully custom `contentEditable` widget with a floating suggestion listbox: it implements Arrow Up/Down + Enter + Escape but there is no visible `role="listbox"`/`role="option"`/`aria-activedescendant` wiring in this file — the active-item highlight is purely visual (`data-active` + background color), so screen-reader users would not get the same affordance as sighted keyboard users.
- `MentionBadge`'s hover tooltip is mouse-only (`onMouseEnter`/`onMouseLeave`); there's no focus-triggered equivalent, so the name/email tooltip is unreachable via keyboard-only navigation.
- Icon-only buttons (document view/download/remove, tag-add) rely on the `title` attribute for a tooltip and implicit accessible name; there is no explicit `aria-label`, so the accessible name depends entirely on `title` text being read by the assistive tech in use.
- Disabled states are consistently paired with either a `disabled` attribute (buttons) or `opacity-60`/`cursor-not-allowed` styling (questionnaire inputs, radio/checkbox labels) — visually clear, and native `disabled` means it's also keyboard/AT-correct for form controls.
- Color is used as a primary signal in several places without a redundant text/icon cue for colorblind users: the rejected/approved header banners do pair color with an icon and text (good), but the WorkflowStepper's "current vs upcoming" distinction relies on border weight + opacity rather than color alone (also good) — no major color-only-signal violations found in this file, but the tag chips' auto-contrast text color is the only accessibility-driven color computation in the file.

## Data Density

This is the densest screen type in the app's idea lifecycle: it simultaneously surfaces the stage stepper, structured questionnaire answers per stage, financial figures, attachments, submission metadata, private manager to-dos, a live activity feed, a separate threaded-remarks history, mention-augmented text everywhere, and points ledger — nine distinct data surfaces on one route. Density is managed by:
- Deferring most content behind tabs (`Details`/`Attachments`, `Activity`/`Remarks`) rather than showing everything at once.
- Deferring two more sections behind native `<details>` disclosures, both closed-by-default-capable (though both ship `open` by default here).
- Fixed max-height scroll regions (`max-h-[360px]`) for the two feed-like lists so they don't push page length, at the cost of a nested scroll area inside an already-scrolling page.
- Loading states are per-query (documents, tags, employees, evaluations, activities each have independent `isLoading`), so a slow one doesn't block the others — but this also means the page can render in a visually "half-populated" state for a moment after the initial skeleton clears.

## Reusable Ideas

- The `InfoRow` helper (label + right-aligned value, `py-2.5 border-b border-border`) is defined locally in this file but is generic enough to belong in the shared component catalog — it duplicates a pattern that recurs in read-only detail rows across the app.
- `tagTextColor()` (luminance-based text contrast for a given background hex) is another file-local utility that is a strong candidate for promotion to a shared color-utils module — any screen rendering user-colored tag chips needs the same computation.
- The imperative-`ref.save()` pattern used for `DynamicStageQuestionnaires` (parent owns the trigger button, child owns the form state and exposes `save()`/`onDirtyChange`/`onSavingChange`) is a reusable shape for "toolbar button controls a form buried in content" that could be documented as a named pattern for future screens with the same save-button-in-header layout.
- The `@[id:Name]` mention-token serialization format and its paired split regex (`/(@\[[^\]]+\])/g`) is duplicated verbatim in three places in this file alone (remark composer output, remarks-tab rendering, and — inside `ActivityFeedRow` — activity descriptions). A shared `parseMentions(text)` helper returning React nodes would remove this repetition.

## Notable Deviations / Debt

- **Legacy import paths, confirmed**: `IdeaDetailScreen.tsx` imports `MentionInput` from `@/components/MentionInput` and `EmployeeIdeasDialog` from `@/components/modal/EmployeeIdeasDialog` — both under the old `src/components/*` tree, not `src/shared/components/*`. This matches the prior flag that this file is one of the few screens still reaching into legacy import paths. The legacy dependency is not superficial: `MentionInput` is the only @-mention composer implementation in the app (feature-local code re-implements only the *display* side, `MentionBadge`, not the input side), so this legacy component is load-bearing, not dead weight.
- The legacy-path pattern recurs one hop downstream: `UploadDocumentModal.tsx` (`@/components/ui` for `Button`) and `IdeaTodoCard.tsx` (`@/components/ui/Icons`, `@/components/shared/AssignDropdown`) — both feature-local components rendered directly by this screen — also reach into `src/components/*`.
- Indigo accent color (`accent-indigo-600`, `bg-indigo-50`, `border-indigo-400`, `text-indigo-600`) is used throughout the questionnaire inputs, file-input styling, and upload dropzone hover states. `12_DESIGN_TOKENS/colors.json` explicitly lists `indigo` under `doNotUse.stockPaletteFamilies`, noting it is "present in a few legacy/secondary screens as tech debt — not part of the canonical [palette]." This screen is one of those legacy/secondary screens for that specific violation.
- Padding/margin values are frequently arbitrary pixel literals (`p-[22px]`, `px-[18px]`, `p-[5px_20px]`, `mb-[10px]`) rather than the standard Tailwind spacing scale steps cataloged in `12_DESIGN_TOKENS/spacing.json`, suggesting this screen was hand-tuned rather than built from the token scale.

## Privacy Rule: `hideSubmitter` (management must never see submitter identity)

**This screen implements the rule, but not via the shared `hideSubmitter` convention used elsewhere.**

- On the Kanban board (`src/features/ideas/container/KanbanScreen.tsx:923`) and the list view (`src/features/ideas/components/IdeaListView.tsx`), the rule is implemented as an explicit, named, passed-down prop: `const hideSubmitter = role === "management";` then threaded through as `hideSubmitter={hideSubmitter}` to card/row sub-components.
- On `IdeaDetailScreen.tsx`, there is no `hideSubmitter` variable at all. The equivalent gating is done **inline**, twice, as the inverse condition `role !== "management"`:
  - **Line ~539**: `{role !== "management" && (<>...</>)}` wraps the entire "Submitted by" `InfoRow` (submitter name, clickable through to the employee profile, plus an edit-submitter pencil icon) **and** the "Assigned Innovation Manager" `InfoRow` (name + reassign pencil icon) inside the "Submission Information" panel. When `role === "management"`, both rows are omitted outright — not blurred, not shown as "Hidden," simply not rendered. The `Reference`, `Submitted` (date), `Status`, and `Tag` rows in the same panel remain visible to management.
  - **Line ~946**: `role !== "management" && idea` gates the entire header `actions` cluster (Reject Idea / Approve Idea / Ready for Pitch, or the rejected/approved status banners). This is a broader restriction than submitter-identity-hiding — it also removes management's ability to take idea-moving actions from this screen, which is a related but distinct access rule bundled into the same conditional.
- **Consistency gap for future maintainers**: because this screen re-derives the same boolean inline instead of importing/naming it `hideSubmitter`, a future refactor of the privacy rule (e.g., extending it to more roles, or centralizing it in a hook like `useHideSubmitter()`) would need to know to also touch this file's two inline `role !== "management"` checks — grepping for `hideSubmitter` alone would miss both. No `hideSubmitter` string exists anywhere in this file.
- **No leakage found**: `idea.submitterName`, `idea.submittedBy`, and `idea.user` are only read inside the gated block; the activity feed (`ActivityFeedRow`) shows `actorName` for all activity types (including presumably submission activity) regardless of role — this is a potential secondary leak path worth flagging for product/backend review (this screen doesn't gate the activity feed's `actorName` by role at all), though whether the backend's `activity` payload for idea-submission events actually contains the submitter's name (as opposed to a generic "System" actor) was not verified from the frontend code alone.
